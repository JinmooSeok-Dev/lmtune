### Case 1-A: FIM 자동완성

#### 시나리오 & 동작

```
사용자: VSCode에서 함수 타이핑 중
  ↓ 300ms debounce
IDE: FIM 요청 자동 전송 (/v1/completions)
  ↓ prefix + suffix → middle 채우기
IDE: 회색 글씨로 제안 → Tab 수락 or 무시
```

- **누가 요청**: IDE가 자동 (사용자 무관)
- **Endpoint**: `/v1/completions` (Chat이 아님)
- **빈도**: 유저당 1–5 req/s → 10명이면 10–50 QPS

#### Multi-turn 특성

| 항목 | 값 | 근거 |
|------|-----|------|
| 턴 수 | **1** (완전히 독립된 단발 요청) | Tab은 stateless FIM 요청 [S-7] |
| 히스토리 | 없음 — 매 요청이 새로운 prefix/suffix | FIM "small snippet" 전송 [S-19] |
| KV Cache 재사용 | **Prefix Caching**: 같은 파일 편집 시 prefix 동일 | (추정) 동일 파일 편집 시 prefix 재사용 유추 |
| Context 누적 | 없음 | stateless 단일 호출 [S-7][S-19] |
| Session 개념 | 없음 (stateless) | 1M+ QPS stateless 요청 [S-19] |

#### I/O 프로파일

| 항목 | 값 | 근거 |
|------|-----|------|
| Input | 1–4K tok (추정), 1–2K tok (추정) | Cursor Tab: context 13K [S-7], "small snippet" 전송 [S-19]. Copilot inline: ~2K context (추정) |
| Output | 10–50 tok (추정) | Cursor Fusion "10x longer changes" [S-7]; FIM output은 짧음, Continue.dev default maxTokens=128 [S-19] |
| I:O | 20:1–100:1 | 위 수치에서 도출 |
| TTFT | **~100ms 개선** [S-1], **p50 260ms** [S-7], **< 200ms** [S-2] | Cursor Fusion p50 475→260ms [S-7], Copilot sub-200ms [S-2] |
| ITL | < 30ms (추정) | 직접 출처 없음, Decode가 극히 짧아 추정 |

#### 추천 데이터셋

### Case 1-E: Thinking 추론 (Extended Thinking) *(신규)*

#### 시나리오 & 동작

```
사용자: "이 알고리즘의 시간복잡도 분석하고 최적화해줘"
  ↓
모델: <think>내부 추론 과정... (2K–8K tokens, 사용자 안 보임)</think>
      최적화된 코드 + 설명 (500–2K tokens, 사용자에게 표시)
```

- **핵심**: 사용자에게 보이는 output과 **실제 GPU가 생성하는 token 수가 2–5배 차이**
- M2.1의 `<think>` interleaved thinking이 대표적
- **Throughput 측정 시 thinking tokens을 포함해야 실제 GPU 부하를 반영**

#### Multi-turn 특성

| 항목 | 값 | 근거 |
|------|-----|------|
| 턴 수 | **1–3** | (추정) |
| 히스토리 | 이전 턴의 thinking content 보존 필요 | (추정) 일반론에서 유추 |
| Context 누적 | **급격히 증가** — thinking tokens이 히스토리에 포함되면 턴마다 증가 | (추정) thinking 비율 직접 출처 없음 |
| KV Cache | thinking 포함 히스토리의 prefix 재사용 중요 | (추정) |

#### 기존 케이스와의 핵심 차이

| 축 | 일반 Chat (1-C 등) | 1-E (Thinking) |
|---|:---:|:---:|
| 실제 생성 tokens | = 사용자 보는 tokens | **사용자 보는 것의 2–5배** |
| 히스토리 크기 | 답변 길이만큼 | **thinking 포함 시 5배로 팽창** |
| Throughput 계산 | output_tokens/s | **(answer + thinking)_tokens/s** |
| Context 압축 | 덜 급함 | thinking 히스토리 보존 vs 압축 트레이드오프 |

### 파라미터 설계 원칙

```
[파라미터 결정 흐름]

1. Input 크기 결정
   ├─ Stateless (1-A~1-C): 실 서비스 per-request 크기 → 직접 대입
   ├─ Multi-turn (2-A~2-B): Turn N까지 누적 median → context window 내 범위
   └─ Agent (3-A, 3-D):    초기 overhead [S-11] + 턴 누적 → window 상한 [S-18]

2. Output 크기 결정
   ├─ FIM (1-A):   10–50 tok (추정) → 50
   ├─ Chat (1-B~2-A): 100–500 (추정) → 300–1500
   ├─ Code Gen (1-C~2-B): 500–3K (추정) → 500–3000
   └─ Agent tool call (3-A): 100–500/턴 (추정) → 3000 (누적 반영)

3. Concurrency 결정
   ├─ Tab completion: 128 (Cursor 1M+ QPS [S-19] → 높은 동시성)
   ├─ Chat/Agent: 4–16 (사용자 대기 중 → 낮은 동시성)
   └─ Multi-Agent (3-D): 100 (8 agents [S-20] × 확장 계수)

4. Multi-turn 설정
   ├─ Short session (2-A): 6턴 (Copilot Chat 패턴)
   ├─ Long session (2-B): 8턴 (멀티파일 편집)
   └─ Agent loop (3-A, 3-D): 30턴 (SWE-bench median, max 150 [S-10])
```

> ⚠️ `--random-input-len` / `--random-output-len`은 **균등 분포** 생성. 실 프로덕션은 Zipf(input) [S-15] / Bimodal(output) [S-15] 분포. 정밀 시뮬레이션 시 BurstGPT [S-15] 또는 ServeGen [S-16] 분포 사용 권장.
